Provably: Safe LLM Generated Proofs
Proof Generation with Lean 4 Verification

A closed-loop pipeline that translates natural-language theorems into machine-checked Lean 4 proofs using large language models, with dual-phase correctness guarantees and self-correcting error feedback.

Chris Rae Andrew Wright School of Informatics & School of Mathematics
Abstract
Overview

Large language models frequently produce confident yet incorrect mathematical proofs, particularly when faced with challenging problems. These solutions often contain subtle hallucinations that are presented as valid reasoning a limitation shared across all current language models.

We present Provably, a verification tool that models can call to check whether their natural-language proofs are correct. Provably uses the lean4 proof assistant to check if a given natural language proof matches a claim. We provide a pipeline to generate the lean and guarantee similarity between the lean code and the provided NL proof.

We also curated a benchmark dataset 136 university-level mathematics problems obtained from The University of Edinburgh's undergraduate maths degree.

Motivation
Bridging the Formalisation Gap

"A proof is only a proof when the compiler accepts it."

Language models excel at writing proofs but routinely hallucinate logically invalid steps. Without formal verification there is no principled way to separate a correct proof from a convincing failure.

Existing tools require users to write Lean directly or produce unverified output. Provably closes this gap automatically.

  • No Lean expertise required — accessible to any mathematician
  • Machine-checkable output guaranteed on every accepted proof
  • Modular back-ends — swap NL or Lean models freely
  • REST API and MCP server for tool integration
Background
Lean 4 & Mathlib

Lean 4 is a dependently-typed proof assistant from Microsoft Research. Mathlib provides over 175,000 formalised lemmas covering undergraduate and graduate mathematics — the formal foundation Provably targets.

All generated proofs use the standard preamble — import Mathlib, import Aesop, set_option maxHeartbeats 0 — giving full tactic access and unrestricted proof search depth.

Verification Strategy
Dual-Phase Correctness

Both phases must pass simultaneously — this prevents syntactically valid but semantically wrong proofs, a failure class compile-only checks miss.

🔨
Phase 1 — Syntactic
lake env lean compiles the file. Rejected if exit code is non-zero or sorry appears anywhere in the source.
+
🔍
Phase 2 — Semantic
An LLM is prompted via outlines, constraining generation to a boolean. It judges whether the Lean proof faithfully implements the original natural-language argument.
Example Verified Proof
Lean 4 — Binomial Sum Identity

Claim: ∑t=0n−k C(n+t, t) = ∑t=kn C(n+t−k, n), for n ∈ ℕ and k ≤ n.

import Mathlib
set_option maxHeartbeats 0
open BigOperators
theorem binomial_sum_identity (n k : ℕ) (h : k ≤ n) :
    ∑ t ∈ Finset.range (n - k + 1), (n + t).choose t
  = ∑ t ∈ Finset.Ico k (n + 1), (n + t - k).choose n := by
  simp [Nat.choose_symm_add, Finset.sum_Ico_eq_sum_range]
  ring

Conclusion & Future Work
Key Findings
  • Closed-loop verification substantially reduces hallucinated proofs — the compiler is a hard correctness filter prompt-engineering cannot bypass.
  • Error feedback as context enables self-correction within three attempts without any model fine-tuning.
  • Modular architecture lets Provably adopt stronger models as they emerge with no changes to the surrounding pipeline.
